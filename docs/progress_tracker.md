## Completed Milestones

### Post-9.0 bug fixes — June 2026
- [x] **PayDialog AEAT timeout now records `PENDIENTE`, not `Error` (and recovery is scoped so it can't mis-resubmit)**: when the 5 s bounded wait expired, `PayDialog::persistPayment` ran a `NETWORK_ERROR` result through `updateTicketVerifactuFields`, which wrote `verifactu_estado = Error`. A slow-but-accepted AEAT submission was therefore recorded as a failure, and `PendingSubmitsDialog` (which only re-surfaces `NULL`/empty/`PENDIENTE`) never offered to reconcile it, so a later manual retry risked an AEAT 8003 duplicate. Now a `NETWORK_ERROR`/`PENDING` outcome (timeout or transport failure — an *unknown*, not a definitive rejection) is persisted as `PENDIENTE` via the new `PayDialog::markPendingVerifactu(seq)`, which keeps the `<n_recibo>-<seq>` InvoiceID and leaves CSV/XML empty; a genuine AEAT `ERROR` reply still writes `Error`. Because `PendingSubmitsDialog` re-submits a whole ticket as a bare `n_recibo` (it is seq-unaware), its query was scoped with `AND verifactu_invoice_seq = 0` so a partial-pay (`seq>0`) `PENDIENTE` row is never auto-retried with the wrong amount/InvoiceID — full seq-aware recovery is filed as a new Open Non-Blocking item. Income totals are unaffected (`PENDIENTE`, like `Error`, is still counted). Files: `src/recog_prendas/pay_dialog.h/.cpp`, `src/app/pendingsubmitsdialog.cpp`, `docs/modules/verifactu/README.md`.
- [x] **Reprinted factura QR now uses the submitted payment date (`fecha_pago`), not `fecha_recepcion`**: `Imprimir::resolveQrCode()` rebuilt the QR's `FechaExpedicion` from row 0's `fecha_recepcion`. For a partial payment (PayDialog submits to AEAT with `fecha_pago`) or a late pickup payment (retry submits with the payment date), payment day ≠ reception day, so a reprint produced a QR whose date — and thus the AEAT cotejo URL — did not match the registered invoice. Now reads the first paid row's `fecha_pago`, which equals the date submitted to AEAT on every path: PayDialog/retry submit the payment date, and save-time submissions store `fecha_pago = fecha_recepcion` (mainwindow binds `:fecha_pago` to `de_date_recep`), so the common counter-sale case is unchanged. Scans for the first non-empty `fecha_pago` because row 0 can be an unpaid garment in the legacy unfiltered reprint path. Files: `src/imprimir/imprimir.cpp`. (Note: the QR is regenerated via the AEAT GetQrCode service rather than rendered from a stored URL because there is no local QR encoder; storing/rendering the cotejo URL locally remains a possible future simplification.)
- [x] **Trimestral accounting lock now reads the whole quarter, not just its last month**: `Contabilidad::on_bb_ok_cancel_accepted()` decided whether a quarter was already done/locked by calling `readLockForMonthAndYear(db, "ingresos", sb_trim*3, year)` — i.e. only the quarter's third month. A quarter with income in its first month(s) but nothing paid in the last month read back as "no data" (status 2), hit the default branch ("no hay registros"), and `updateLock()` was never invoked, so the quarter stayed editable after the operator closed contabilidad — AEAT-reported income could then be altered. New `sql_lite::readLockForQuarter(db, table, quarter, year)` does one pass over the quarter's three months (`substr(dateCol,4,2) IN (m1,m2,m3) AND substr(dateCol,7,4)=year`): `COUNT(*)=0` → 2 (no data), else `MAX(edit_lock)` → 1 locked / 0 open. Used both for the lock decision and for the Annual report's per-quarter "Contabilidad cerrada/no cerrada" label (which had the same last-month-only bug). As part of the change the lock read moved inside the `Trimestral` branch (it was computed unconditionally at the top, producing a meaningless `month=sb_trim*3` up to 36 in Mensual mode) and the dialog's stay-open-on-revert decision now uses an explicit `keepDialogOpen` flag instead of re-testing the now-scoped `editLock`. Files: `src/sql_lite/sql_lite.h/.cpp`, `src/contabilidad/contabilidad.cpp`, `docs/modules/{contabilidad,sql_lite}.md`.
