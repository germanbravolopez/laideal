- **Archive** — entries older than ~6 months and no longer actionable.

## Completed Milestones

### Post-10.3 development — July 2026
- [x] **Updater downloads the installer to a per-user app dir, not `%TEMP%` (the real auto-update blocker)**: 10.3's `ShellExecute("runas")` (elevation) was necessary but insufficient - on-machine testing of the 10.2->10.3 update showed the installer downloaded fine but **could not be launched at all**, not by the app and not even by double-clicking it in Explorer ("Windows cannot access the specified device, path, or file... permissions"). Diagnosed on the shop PC: the file's inherited ACL in `%TEMP%` was `Administrators:(F)` + `Everyone:(R,W,DC)` with **no execute for the normal user** - a hardened Temp (the "AV/Controlled-Folder blocks executing from %TEMP%" the README warns about, implemented as an ACL that omits execute). Under UAC a non-elevated process has the Administrators SID as deny-only, so it fell to `Everyone` = read/write but not execute -> access denied; running the app as admin masked it because the Administrators ACE then applied. Confirmed the same file copied to `%LOCALAPPDATA%\<app>\updates` gets `<user>:(F)` (execute allowed). Fix: `Updater::downloadInstaller` now saves to `QStandardPaths::AppLocalDataLocation + "/updates"` (per-user, execute-allowed) instead of `TempLocation`, and clears stale `*.exe` there first so installers don't pile up. Combined with 10.3's `ShellExecute("runas")`, the update now downloads to an executable per-user folder -> raises the UAC prompt -> installs into Program Files, all from a normally-launched app. **Note:** every build up to 10.3 has the broken (Temp) updater, so the fixed version must be installed **manually once**; auto-update works from that version onward. Zero-UAC alternative (per-user install, "Fix B") still filed as Open Non-Blocking. **Not unit-tested**: environment-dependent download-path + Win32 launch, no pure logic; `compareVersions` stays covered. Files: `src/updater/updater.cpp`, docs.

### Post-10.2 development — July 2026
- [x] **In-app updater no longer requires launching the app as administrator (updater "Fix A")**: investigated from a shop report that the auto-update only worked when the app was started with admin rights. Root cause chain: (1) the installer targets `C:\Program Files\La Ideal` (`releases/laideal.iss` uses `DefaultDirName={autopf}` and no `PrivilegesRequired`, so Inno defaults to **admin**), so `laideal_setup_X.Y.exe` is manifested `requireAdministrator`; (2) `UpdaterDialog::onDownloadFinished` launched it with `QProcess::startDetached`, which uses Win32 `CreateProcess` - and `CreateProcess` **cannot** elevate, so launching an admin-required installer from a non-elevated app fails with `ERROR_ELEVATION_REQUIRED` and `startDetached` returns false (the "no se pudo iniciar el instalador" error). When the app itself was elevated, the elevated parent launched the elevated child fine - hence "run as admin". Fix: on Windows the installer is now launched via `ShellExecuteW(nullptr, L"runas", ...)`, which raises the UAC prompt so a normally-started app can run the (elevated) installer; the non-Windows path keeps `QProcess::startDetached`. `shell32` added to the `updater` link on WIN32. **Not unit-tested**: the change is a single Win32 elevated-process launch (UAC), pure integration with no extracted logic; the updater's testable core (`compareVersions`) stays covered by `test_versioncompare`. The zero-UAC alternative (a per-user install, "Fix B") is filed as an Open Non-Blocking item. Files: `src/updater/updaterdialog.cpp`, `src/updater/CMakeLists.txt`, docs.
