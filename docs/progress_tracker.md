### 8.4 release — May 2026
- [x] **Split `verifactutypes.h` out of `verifactumanager.h`**: the previous milestone moved `updateTicketVerifactuFields()` into `sql_lite.cpp` and added `#include "../verifactu/verifactumanager.h"` to bring in `VerifactuResult` + `VerifactuEstado` + the helpers. That header transitively pulls `<QNetworkAccessManager>` (because it declares the REST client), so the `sql_lite` static lib - which links only Core / Gui / Sql, not Network - failed to compile (`fatal error: QNetworkAccessManager: No such file or directory`). Lifted the lightweight types (the `VerifactuEstado` enum, `verifactuEstadoToString/FromString`, the `VerifactuResult` POD) into a new `verifactu/verifactutypes.h` that depends only on `<QString>` + `<QPixmap>`. `verifactumanager.h` now includes that header instead of carrying the types; `sql_lite.cpp` includes the small header so it never sees Network. Same one-class-per-header direction as the open Tier-1 task for `verifactuinvoice.h`. Files touched: `src/verifactu/verifactutypes.h` (new), `src/verifactu/verifactumanager.h`, `src/verifactu/CMakeLists.txt`, `src/sql_lite/sql_lite.cpp`.
- [x] **`RecogPrendas`: fix latent proxy/source row-coord bug under header sort + set `table_name` so date / `importe` sort correctly**: `recog_prendas.ui` already had `sortingEnabled=true` on the table view and `on_pb_search_clicked` already called `sortByColumn(0, DescendingOrder)`, so header-click sorting was nominally working - but two flaws made it silently wrong. **(1) Coordinate-space bug**: `on_tableView_clicked(const QModelIndex &index)` received a proxy-space index from the view, stored `index.row()` as `rowClickedCell`, and the rest of the class then consumed that value as a *source-model* row via `sqlQueryModel->index(rowClickedCell, ...)`. Whenever the desc-by-n_recibo default sort actually reordered rows (i.e. searches that span multiple tickets - typically name search), every click that wasn't on the first visible row hit the wrong DB record - so `pb_state` (Entregar) could mark the wrong ticket as picked up, `pb_separ_garm` could split the wrong garment, etc. Same-ticket searches were immune because all rows share the same n_recibo so the sort was a no-op. Factored row storage into a new private helper `selectSourceRow(int sourceRow, int sourceCol)`; `on_tableView_clicked` now maps via `proxyModel->mapToSource(index)` before delegating. The pay-all / pku-all loops, which previously fake-called `on_tableView_clicked` with raw `sqlQueryModel->index(row, 0)` source indices (an existing layering oddity that happened to work because of the bug above), now call `selectSourceRow(row, 0)` directly - same outcome, less magic. **(2) `table_name` was never set on the proxy**: `MySortFilterProxyModel::lessThan()` keys on `table_name` (`"ingresos"` / `"gastos"` / `"prendas"`) to pick the date or numeric comparator; `Listado` sets it but `RecogPrendas` never did, so clicking a date column header sorted lexicographically (`"31-01-2026" < "02-02-2026"` reverses by day, not chronology) and `importe` sorted as strings (`"100" < "9"`). Added `proxyModel->table_name = "ingresos";` right after the proxy is constructed. **Defensive**: `MySortFilterProxyModel *proxyModel` was an uninitialised pointer member; changed to `= nullptr` so the `proxyModel ? mapToSource : index` guard in the click handler is safe before the first search. The `sortByColumn(0, ...)` call now uses `INGRESOS_COL_N_RECIBO` instead of a bare literal. Files touched: `src/recog_prendas/recog_prendas.h/.cpp`.
- [x] **Consolidate `ingresos` column-index constants into `sql_lite/ingresos_schema.h`**: three parallel macro sets indexed the same 24 columns of `ingresos`: `INGRESOS_IDX_*` in `mysortfilterproxymodel.h` (used by `mysortfilterproxymodel.cpp`, `mainwindow.cpp`, `listado.cpp` — half-Spanish names ID/IMPORTE/PAYED/CANTIDAD/PRENDA/SERVICIO/OBSV), `TABLE_*` in `imprimir.h` (used by `imprimir.cpp`, `add_garment.cpp` — English-ish names TICKET/PRICE/IS_PAYED/QUANTITY/GARMENT/SERVICE/OBSERV), and a near-duplicate `TABLE_*` in `recog_prendas.h` (with `TIMESTAMP` instead of `TS` and a comment leaning on imprimir.h for `TABLE_HASH`). The previous Verifactu milestones in the tracker explicitly note new columns being added to all three blocks for each schema change — exactly the drift hazard that produced the v8.0 `verifactu_xml`/`verifactu_hash` silent-drop bug. Lifted into a single canonical set in a new tiny header `sql_lite/ingresos_schema.h` (no QtSql dep, so light consumers like `tableview` can include it without picking up the SQL link): `INGRESOS_COL_<UPPER_DB_COLUMN_NAME>` (`INGRESOS_COL_N_RECIBO=0` ... `INGRESOS_COL_VERIFACTU_RECTIFICATION_TYPE=23`). `sql_lite.h` re-includes it so existing SQL consumers keep working transparently. The name = uppercase SQL column avoids the English-vs-Spanish bikeshed (DB names are what they are) and makes a grep for any column hit both the schema and the C++ constant. All ~80 call sites renamed (mainwindow.cpp, listado.cpp, mysortfilterproxymodel.cpp, imprimir.cpp, recog_prendas.cpp, add_garment.cpp). Headers (`mysortfilterproxymodel.h`, `imprimir.h`, `recog_prendas.h`) gain a one-line breadcrumb pointing at `sql_lite.h` so future readers know where the constants live. `TABLE_TICKET_*` in `mainwindow.h` was **not** touched — those index the in-memory `QTableWidget` used for ticket entry, a distinct concept from the `ingresos` DB row layout. The mechanical rename was driven by a regex script with `\b` boundaries so `TABLE_TICKET_QNTY` etc. were correctly skipped. Docs updated: `docs/modules/imprimir.md`, `docs/modules/recog_prendas.md` (column-index tables rewritten to point at `INGRESOS_COL_*`). Files touched: `src/sql_lite/sql_lite.h`, `src/tableview/mysortfilterproxymodel.h`, `src/imprimir/imprimir.h`, `src/recog_prendas/recog_prendas.h`, `src/app/mainwindow.cpp`, `src/listado/listado.cpp`, `src/tableview/mysortfilterproxymodel.cpp`, `src/imprimir/imprimir.cpp`, `src/recog_prendas/recog_prendas.cpp`, `src/add_garment/add_garment.cpp`, `docs/modules/imprimir.md`, `docs/modules/recog_prendas.md`.

- [x] **Save-time print: bounded 3 s wait for AEAT so the factura carries the QR when possible, otherwise degrade to a paid recibo**: previously the `isPaid` branch of `on_bb_save_reset_clicked` always called `printFra()` immediately after firing the async `verifactuSubmitInvoice()`, so the printed factura simplificada always went out without QR / CSV - the AEAT reply was still in flight and the DB row was still PENDIENTE. Otherwise (timeout, success-without-QR, or error) we call `printRecibo()`, which - because the ticket is paid - already stamps "Importe pagado" on the customer copy via `createTicketExcel(true, ui->pb_payment->isChecked())`. The async submit is still alive in `m_pendingSubmits` and `onVerifactuRequestFinished` will eventually patch the row even if it missed the 3 s window; the customer can reprint a Verifactu-complete copy via RecogPrendas at pickup time once `ENVIADA` lands. Picked 3 s because the median Irene Solutions round-trip from local logs is around 1-2 s; longer waits start to feel laggy at the counter. Files touched: `src/app/mainwindow.h/.cpp` - new `+#include <QEventLoop>`, `verifactuSubmitInvoice` signature change (`void` → `QString`), `printFra` gains a default `QPixmap` parameter, save handler rewritten with the bounded wait. `docs/architecture.md` Data Flow updated.

- [x] **`RecogPrendas` name search silently dropped tickets past row 256** (no log was registered because the function had no entry-time `qDebug`): the name-search branch did `db.open() / SELECT * FROM ingresos / setQuery() / db.close()` and then, ~30 lines later in the common post-search section, ran `while (sqlQueryModel->canFetchMore()) sqlQueryModel->fetchMore();` to drain the rest.
