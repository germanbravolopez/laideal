## Completed Milestones

### Post-8.0 maintenance тАФ May 2026
- [x] **`RecogPrendas` "Pagar todo" now auto-prints the factura and offers a second copy; print no longer closes the window**: previously the operator had to click "Pagar todo" (which marks every row of the ticket as paid and fires the async Verifactu submission), then separately click the print button to send the factura to the thermal printer. The print button also unconditionally closed the RecogPrendas window, which made it awkward to print a second copy for the establishment - the operator had to reopen RecogPrendas, search the ticket again and reprint. New flow: at the end of `on_pb_pay_all_clicked()`, the ticket number is captured **before** `resetAllContents()` clears `le_nr_ticket` (when no row was originally clicked the loop leaves it on the last paid row), and the factura is generated + sent to the printer via a new private helper `printFactura(ticketNum, askSecondCopy)`. When `askSecondCopy=true`, the helper then opens a `QMessageBox::question` "┬┐Imprimir una segunda copia de la factura?" (Yes / No, defaulting to No) and re-runs `printTicket()` on the same Excel file if the operator picks Yes. After the print returns, `resetAllContents()` clears the form so the operator can move on to the next client without reopening the window. `on_pb_print_clicked()` is refactored to call the same helper with `askSecondCopy=false` and then `resetAllContents()` (replacing the previous `this->close()`, so the standalone print button no longer dismisses the window either). Both copies use the same Excel parameters (`createTicketExcel(copyForClient=false, addPayedInfo=false)`) so the output is identical - "second copy" just reprints the same file rather than rendering a different variant. The `AppSettings::enablePrinting()` guard wraps both the first print and the second-copy dialog, so testing-mode runs still generate the Excel but skip the printer and the dialog. Caveat: when "Pagar todo" is pressed on a multi-ticket search (e.g. name search returning all of a client's tickets), only the last paid ticket gets printed - separate `printFactura` calls per `n_recibo` would be needed to cover that, deferred until the multi-ticket case is actually exercised. A `qDebug` was added at the top of `printFactura` showing `ticket` and `askSecondCopy` so the operator can confirm the flow ran via `~/.laideal.log`. Files touched: `src/recog_prendas/recog_prendas.h/.cpp`.
- [x] **`RecogPrendas` name search silently dropped tickets past row 256** (no log was registered because the function had no entry-time `qDebug`): the name-search branch did `db.open() / SELECT * FROM ingresos / setQuery() / db.close()` and then, ~30 lines later in the common post-search section, ran `while (sqlQueryModel->canFetchMore()) sqlQueryModel->fetchMore();` to drain the rest. But `QSqlQueryModel` lazy-fetches in 256-row batches and `fetchMore()` needs the underlying SQLite statement handle to still be valid, which `db.close()` had already invalidated. So `canFetchMore()` returned `false` immediately and the proxy filter only ever saw the first 256 rows of `ingresos` - any client whose receipts landed past that point would have some tickets in the result and some silently missing, regardless of whether the search term was a partial or complete name. Fix: pulled the `fetchMore()` loop **inside** the name branch, ordered as `setQuery -> fetchMore-until-done -> db.close()`, and dropped the now-redundant common loop. Added three `qDebug` lines so the search is now visible in `~/.laideal.log`: (1) at function entry, showing the input and the date-mode combobox value; (2) after the SELECT, showing how many source rows were loaded and the normalized filter; (3) at the end, showing `proxyMatched / sourceLoaded`. The same `db.close()` pattern exists in the phone / ticket / date branches but doesn't trigger the bug there because each of those returns a small result set (single client, single ticket, single day) that fits in one 256-row batch. Files touched: `src/recog_prendas/recog_prendas.cpp`.
- [x] **Streamlined release pipeline: one PowerShell command builds + deploys + installs**: the previous flow required opening the Qt Creator IDE, building Release manually, closing Qt Creator, opening a special "Qt command prompt" shortcut (`cmd.exe /K C:\Qt\6.4.3\mingw_64\bin\qtenv2.bat`) with admin rights, and running `deploy_laideal_run_in_qt_cmd.bat` which chained three other batch scripts (`zip_folder.bat`, `find_replace.bat`, `create_installer.bat`). The deploy script searched for Qt Creator's specific shadow-build folder (`build-laideal-Desktop_Qt_*_MinGW_64_bit-Release`), so the README's `cmake -B build` instructions actually produced an exe in a folder the deploy script could not find - a hidden disconnect that broke the documented build path. Replaced by a single `releases\release.ps1 <version>` invocation (with `releases\release.bat <version>` as a thin cmd.exe wrapper) that: (1) prepends Qt + MinGW to `PATH` itself so the Qt cmd shortcut is no longer required, (2) cross-checks the supplied version against `project(laideal VERSION X.Y ...)` in `CMakeLists.txt` and aborts on mismatch, (3) refuses to overwrite an existing zip/installer/staging dir, (4) configures CMake (Release, MinGW Makefiles) into `build-release/`, (5) builds with `-j`, (6) stages the exe into `releases\<version>\` and runs `windeployqt --release --no-translations --no-system-d3d-compiler`, (7) zips into `releases\old_releases\<version>.zip` via .NET `System.IO.Compression.FileSystem`, (8) compiles the Inno Setup installer via `ISCC.exe /DMyAppVersion=<version> laideal.iss`, (9) deletes the staging dir. The `laideal.iss` template was rewritten to use ISCC preprocessor `#define MyAppVersion` (with `#ifndef` guard so it still compiles standalone with a 0.0 default), and all paths inside it switched from absolute to relative-to-iss (`..\License.txt`, `..\releases_notes.txt`, `..\icon\lavadora.ico`, `setup_outputs`, `{#MyAppVersion}\*`) so the installer recipe is portable to any clone of the repo, not just this developer's machine. The four obsolete batch scripts were deleted (`deploy_laideal_run_in_qt_cmd.bat`, `zip_folder.bat`, `find_replace.bat`, `create_installer.bat`) - the brittle `find_replace.bat` sed-replace of "1.0" in the iss is now redundant since ISCC handles substitution natively. `.gitignore` updated: dropped the now-unused `laideal_upd.iss` entry, added `build-release` alongside `build`. README's "Build" section now shows the explicit `-G "MinGW Makefiles" -DCMAKE_BUILD_TYPE=Release` flags (so a fresh CMake configure picks the right toolchain even on machines with Visual Studio installed), and the "Release procedure" collapsed from 12 manual steps to 5 (bump version + commit, run script, tag, push, gh release). Files touched: `releases/release.ps1` (new), `releases/release.bat` (new), `releases/laideal.iss` (rewritten), `.gitignore`, `README.md`; deleted `releases/deploy_laideal_run_in_qt_cmd.bat`, `releases/zip_folder.bat`, `releases/find_replace.bat`, `releases/create_installer.bat`.
